--- EXAMPLE 1 ---
ID: practice_0000
Type: conversation

Scenario: ALAN practices and rehearses newly learned knowledge

USER: User teaches ALAN about database sharding

ASSISTANT: So sharding splits a database across multiple servers by distributing rows based on a shard key — like splitting users A-M on server 1 and N-Z on server 2?

ASSISTANT: If I apply this to the e-commerce scenario you mentioned: we could shard the orders table by customer_id, so each shard handles a subset of customers. Queries for a specific customer hit only one shard.

ASSISTANT: This reminds me of the map-reduce pattern — both distribute data across nodes for parallel processing. Sharding for storage, map-reduce for computation.

ASSISTANT: What happens when you need to query across shards — like finding the total number of orders across all customers? Is that where a coordinator node comes in?

ASSISTANT: So the core idea is: instead of one server handling all the data, you split the data across servers using a key. Each server handles its portion independently. The tradeoff is simpler per-shard queries but more complex cross-shard operations.
